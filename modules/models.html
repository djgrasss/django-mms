accounting: Double-Entry Accounting Core¶

class accounting.models.LedgerAccount(*args, **kwargs)¶

A particular account in the accounting ledger system.

All transactions must have a left side (debit) and a right side (credit), and they must add up to zero.

This implements the double-entry bookkeeping system. More background information is available in the Wikipedia:

https://en.wikipedia.org/wiki/Double-entry_bookkeeping_system

The normal balance for each account is determined by the account_type, in particular whether the integer value is less than zero:

Type Int Balance
Expense -2 debit
Asset -1 debit
Equity 0 credit
Liability 1 credit
Income 2 credit
Parameters:
  • gnucash_account – Corresponding Gnucash account. Used for data exchange with the core accounting system.
  • account_type – Type of account, based on double-entry bookkeeping principles. This is from our perspective: an account where a member deposits money would be a Liability account.
account_balance¶

The balance of this account.

Unlike balance, this flips the sign for credit accounts (e.g. Equity, Liability, Income).

>>> acct.account_type is TYPE_LIABILITY
True
>>> acct.debits
Decimal('42.00')
>>> acct.credits
Decimal('69.00')
>>> acct.account_balance
Decimal('-27.00')
balance¶

The raw balance of debits and credits on this account. Simply put, this returns debits minus credits without any regard for the account’s normal balance.

>>> acct.debits
Decimal('42.00')
>>> acct.credits
Decimal('69.00')
>>> acct.balance
Decimal('-27.00')

The summation of balance across all LedgerAccount will (should?) equal exactly zero.

See also: account_balance

credits¶

The sum of all credit transactions on this account.

>>> acct.credits
Decimal('42.00')
debits¶

The sum of all debit transactions on this account.

>>> acct.debits
Decimal('23.23')
get_account_transactions()¶

Returns a django.db.models.query.QuerySet with all LedgerEntry instances referencing this account, whether credit or debit.

Example to print all effective dates and amounts:

>>> for txn in acct.get_account_transactions():
...    print(txn.effective_date, txn.account_net(acct))

account_net(account)¶

Net effect of this transaction on account.

Equation:
amount = debit - credit

This method returns amount if account is the debit (left-hand) account and -amount if account is the credit (right-hand) account.

Parameters:account – Account to compute net effect upon.
class accounting.models.PaymentMethod(*args, **kwargs)¶

A valid method of payment, for adding money to a member’s account.

This provides a mapping between a payment processor’s API and two LedgerAccount instances, one for revenue and another for fees.

Equation:
revenue = payment - fees

where payment is the amount credited to the member’s account.

Parameters:
  • name – User-friendly description of this payment method.
  • api – The API to use on the back end, selected from API_CHOICES
  • is_recurring – True if this method can be used for recurring payments (e.g. Stripe), False if it cannot (e.g. Cash).
  • is_automated – True if this method is completely automated from the end user’s perspective (e.g. Stripe), False if it requires intervention (e.g. Cheque).
  • revenue_account – LedgerAccount for the net transaction amount. Must be TYPE_ASSET.
  • fee_account – LedgerAccount for the transaction fees. Must be TYPE_EXPENSE.

static add_n_months(original, months)¶

Adds (or subtracts) a number of months from the original date.

Tries to get as close to “the *n*th of the month” as possible, but in the event that date doesn’t exist (e.g. February 30th), it will choose a close date. For example, 1 month after January 31, 2015 will be February 28, 2015.
